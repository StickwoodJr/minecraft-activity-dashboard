# Project Overview: Fabric 1.21.1 Server-Side Dashboard Mod

## Context & Goal
You are tasked with porting an existing Python/Flask Minecraft log parsing web application into a native, server-side Minecraft mod for Fabric 1.21.1. 

The target production environment is a hosted server with a strict 10GB total RAM limit. Because the Minecraft server process is memory-intensive, this mod must have a near-zero memory footprint. We are replacing the previous "in-memory cache" architecture with a "disk-backed incremental" architecture.

## Environment & Constraints
- **Modloader:** Fabric 1.21.1.
- **Web Server:** You MUST use Java's built-in `com.sun.net.httpserver.HttpServer`. Do not add external web frameworks like Javalin, Spring, or Spark to avoid bloating the `.jar`.
- **JSON Serialization:** Use Google's `Gson`. Include it in the `build.gradle` so it is shadowed/bundled with the mod.
- **Memory Strictness:** You must not store parsed historical log data in active memory. Write data to a `.json` file on disk, clear variables, and stream the file directly to web clients.

## Architecture Requirements

### 1. Configuration (`dashboard-config.json`)
Create a lightweight configuration file in the server's `config/` directory. It should generate on first boot with default values so it can be quickly edited via Vim later. 
- `web_port`: default `5000`
- `incremental_update_interval_minutes`: default `5`
- `ignored_players`: array (e.g., `["ironfarmbot", "mobfarmbot"]`)

### 2. Disk-Backed Data Cache (`dashboard_cache.json`)
This file will live in the server root or `config/` directory and hold the JSON output (`daily`, `playerDailyRaw`, `sessData`, `hourly`).
- **Historical Run (One-Time):** On server start, if this file does NOT exist, sequentially parse all `.log.gz` and `.log` files in the `logs/` directory. Calculate the activity, generate the JSON string, write it to `dashboard_cache.json`, and immediately garbage collect the parsed data from RAM.
- **Incremental Run:** If the file DOES exist, run a scheduled background thread every `X` minutes (based on config). This thread should only read newly appended lines from `logs/latest.log`, update the specific player session values in `dashboard_cache.json`, save it to disk, and clear RAM.

### 3. Embedded Web Server Thread
- Run the `HttpServer` on a dedicated background thread using a fixed `ExecutorService` (maximum 2 threads) to protect the main Minecraft server TPS.
- Tie the startup and graceful shutdown of this server directly to Fabric's `ServerLifecycleEvents.SERVER_STARTED` and `SERVER_STOPPING` events.

### 4. Endpoints
- `GET /` -> Serve `mc-activity-heatmap-v13.html` directly from the mod's `src/main/resources/web/` directory. Set `Content-Type` to `text/html`.
- `GET /api/activity` -> Read `dashboard_cache.json` from the disk and stream it to the HTTP response output stream. Set `Content-Type` to `application/json`. Do not parse the JSON into Java objects for this step; just pipe the file contents.

## Execution Steps for Gemini CLI
1. Initialize a standard Fabric 1.21.1 mod workspace. Set up the `build.gradle`, `gradle.properties`, and `fabric.mod.json`.
2. Implement the `dashboard-config.json` generator and parser.
3. Translate the core logic of the provided Python regex log parser into a Java utility class. Apply the logic to build the four key structures (`daily`, `playerDailyRaw`, `sessData`, `hourly`).
4. Implement the Historical and Incremental disk-caching logic.
5. Create the embedded Java `HttpServer` class with the `/` and `/api/activity` contexts.
6. Register the lifecycle events in the main Mod Initializer.
7. Place the provided `mc-activity-heatmap-v13.html` into the resources folder.

## Acceptance Criteria
- [ ] The mod compiles successfully via `./gradlew build` without dependency errors.
- [ ] The embedded web server starts on port `5000` when the Minecraft server starts.
- [ ] Navigating to `http://localhost:5000/` loads the HTML file.
- [ ] Navigating to `http://localhost:5000/api/activity` returns valid JSON matching the schema expected by the frontend.
- [ ] The RAM usage of the parser scales smoothly regardless of how many gigabytes of historical logs are present in the `logs/` folder.

- The Regex patterns used to extract timestamps, joins, and leaves.

## MODEL ✨

I have successfully ported the Python/Flask Minecraft log parsing application into a native, server-side Fabric 1.21.1 mod.

### Highlights of the Implementation
- **Mod Initializer (`FabricDashboardMod`)**: Ties the web server and parser startup to the `ServerLifecycleEvents.SERVER_STARTED` and `SERVER_STOPPING` events.
- **Web Server (`DashboardWebServer`)**: Utilizes Java's built-in `com.sun.net.httpserver.HttpServer` on a fixed `ExecutorService` (2 threads) to safely run the background web server without bloating the `.jar` with an external web framework or affecting Minecraft's TPS.
- **Log Parsing (`LogParser`)**: Migrated the core regex parsing from Python. It features two execution modes:
  - **Historical**: Iterates chronologically through all `logs/*.log.gz` and `logs/*.log` files. Instead of holding gigabytes of raw logs in memory, it uses streaming `GZIPInputStream` and `BufferedReader` to process events sequentially, aggregating small amounts of structural JSON data (maps) before immediately writing `dashboard_cache.json` and clearing the references.
  - **Incremental**: Scheduled via `ScheduledExecutorService` (based on the configurable update interval) to selectively tail `latest.log` using a stateful `lastByteOffset` pointer. It parses only newly appended lines since the last tick to gracefully update the cache.
- **Resource Bundling**: `Gson 2.10.1` is bundled via Loom's `include` dependency scope to handle serialization securely. The provided `mc-activity-heatmap-v13.html` has been injected directly into the `src/main/resources/web/` directory and streams raw via `getResourceAsStream()`.

### Codebase Organization
The complete workspace has been structured under `/home/gstickwood/gemini-projects/playtime-mod/fabric-dashboard`:

## MODEL ✨

The issue was caused by an outdated version of the Fabric Loom plugin (`1.8-SNAPSHOT`) in the `build.gradle` file. 

Older versions of the Loom build tool have compatibility issues when trying to package Java 21 and Fabric 1.21.1 projects, which caused it to silently fail during the final remapping step. As a result, the built `fabric-dashboard-1.0.0.jar` was essentially an empty 260-byte file containing only a manifest, completely stripping out the `fabric.mod.json` and all the compiled Java classes! 

Because the JAR lacked the `fabric.mod.json` descriptor, the Fabric loader rejected it as a "non-fabric mod" and never executed the Java code, which is why the configuration file was never generated.

I have updated `fabric-dashboard/build.gradle` to use the latest compatible Loom version (`1.16-SNAPSHOT`) and rebuilt the mod. 

```gradle
plugins {
	id 'fabric-loom' version '1.16-SNAPSHOT'
	id 'maven-publish'
}
```

The newly built `fabric-dashboard/build/libs/fabric-dashboard-1.0.0.jar` is now correctly sized (~350KB) and successfully packages the `fabric.mod.json`, the bundled Gson library, your HTML frontend, and the compiled Java mod. 

If you copy the new `.jar` into your server's `mods/` directory and start the server, the mod will be correctly identified, the web server will start, and `dashboard-config.json` will be generated in your `config/` directory!
